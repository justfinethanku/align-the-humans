## Findings
### [high] [CONFIRMED] Database transition trigger still blocks round-2 re-analysis
File: supabase/migrations/20251110051815_init_human_alignment.sql:414

What happens: After both round-1 resolutions are submitted, `submit-resolution` stores each pick as round-2 input (`responseRound = round + 1` at app/api/alignment/[id]/submit-resolution/route.ts:117, then `saveResponse` at app/api/alignment/[id]/submit-resolution/route.ts:142 and `submitResponse` at app/api/alignment/[id]/submit-resolution/route.ts:165). When both are present, it increments only `current_round` to `responseRound` at app/api/alignment/[id]/submit-resolution/route.ts:219; its own comments say status stays `resolving` at app/api/alignment/[id]/submit-resolution/route.ts:204. The analysis page is then responsible for re-analysis: it sees no current-round analysis, verifies at least two submitted responses, and calls `triggerServerAnalysis()` at app/alignment/[id]/analysis/page.tsx:216 and app/alignment/[id]/analysis/page.tsx:228. The analyze API tries to flip `resolving -> analyzing` at app/api/alignment/analyze/route.ts:157. TypeScript allows that edge at app/lib/types.ts:482, but the installed migration trigger only allows `resolving -> complete` at supabase/migrations/20251110051815_init_human_alignment.sql:414, and the trigger is created at supabase/migrations/20251110051815_init_human_alignment.sql:425. A repo-wide migration search found no later replacement for this trigger.

User-visible effect: The page-guard version of the historic `/resolution <-> /analysis` loop is fixed, but round 2 is still blocked on a database built from the current repo migrations. The user reaches `/analysis`, the API update fails, and no new analysis/document phase is produced.

Fix sketch: Add a migration that replaces `public.validate_alignment_transition()` so it matches `VALID_STATUS_TRANSITIONS`, specifically allowing `resolving -> analyzing`. Then run the round-1 submit-to-round-2-analysis path against a migrated database.

### [high] [CONFIRMED] Analyze errors can strand an alignment in analyzing
File: app/api/alignment/analyze/route.ts:157

What happens: The analyze API changes status to `analyzing` before response fetch, AI generation, analysis insert, or final status restoration at app/api/alignment/analyze/route.ts:157. If anything after that fails, the catch block only returns an error response at app/api/alignment/analyze/route.ts:373 and app/api/alignment/analyze/route.ts:390; it does not roll the alignment back to `active` or `resolving`. Future analyze calls with status `analyzing` only succeed if an analysis row already exists at app/api/alignment/analyze/route.ts:133; otherwise they throw `INVALID_STATUS` at app/api/alignment/analyze/route.ts:149. The analysis page has no polling or in-flight branch: if no analysis exists it calls the API immediately, and throws on any non-OK response at app/alignment/[id]/analysis/page.tsx:216 and app/alignment/[id]/analysis/page.tsx:539.

User-visible effect: A transient AI/save/network failure can strand the record in `analyzing`, after which `/resolution` redirects to `/analysis` and `/analysis` keeps erroring. During simultaneous page loads, the losing request can also hit the same non-polling error while the winning analysis is still in flight.

Fix sketch: Treat `analyzing` as an explicit in-flight state in the UI: poll for the analysis row, show retry/recover actions, and add a stale-lock recovery path that can reset `analyzing` back to `resolving` or safely resume analysis when no analysis row exists.

### [low] [CONFIRMED] First resolution submitter is sent back to analysis instead of the waiting state
File: app/alignment/[id]/resolution/resolution-form.tsx:316

What happens: If the current user submits before the partner, the client pushes `/alignment/${alignmentId}/analysis?waiting=true` at app/alignment/[id]/resolution/resolution-form.tsx:316. `AnalysisPage` does not accept or read `searchParams`; when the existing current-round analysis exists, it renders the report and action button at app/alignment/[id]/analysis/page.tsx:253. The actual submitted/waiting state is on the resolution page when `hasUserSubmitted` is true at app/alignment/[id]/resolution/resolution-form.tsx:383.

User-visible effect: The first submitter lands back on the old analysis report after submitting, then has to click back into resolution to see the waiting screen. This is confusing, but it is not the historic redirect loop.

Fix sketch: After a non-final resolution submission, route back to `/alignment/${alignmentId}/resolution` so the server component can render `hasUserSubmitted`, or make `AnalysisPage` honor `waiting=true` with a real waiting view.
