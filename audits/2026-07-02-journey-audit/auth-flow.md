# auth-flow audit

## Verdict
ISSUES-FOUND

## Findings

### [medium] [CONFIRMED] Token-hash signup confirmation can bounce back through callback and login
File: app/auth/callback/route.ts:120
What happens: The token_hash branch verifies the OTP, then prefers `next` or `redirect_to` before its own fallback. The signup action sets `emailRedirectTo` to `/auth/callback` (`app/(auth)/signup/actions.ts:152`), and the custom email hook embeds the incoming `redirect_to` into signup confirmation links (`app/api/auth/email/route.ts:191` and `app/api/auth/email/route.ts:200`). When that `redirect_to` is `/auth/callback`, the verified callback redirects to `/auth/callback` again (`app/auth/callback/route.ts:120` to `app/auth/callback/route.ts:124`). The second callback request has no `code` or `token_hash`, so it falls to `/login?error=verification_failed` (`app/auth/callback/route.ts:128` to `app/auth/callback/route.ts:132`); if the OTP established a session, middleware then redirects `/login` to `/dashboard` (`middleware.ts:125` to `middleware.ts:127`).
User-visible effect: A just-confirmed signup can take an unnecessary callback -> login -> dashboard path. That is a plausible source of the reported snap/flicker, and if the session is not visible to middleware on the final `/login` request the user can see a false verification-failed message.
Fix sketch: In the token_hash branch, ignore `redirect_to` values that resolve to `/auth/callback` for signup/email_change, or make signup confirmation links carry `redirect_to=/dashboard` or `next=/dashboard`. Keep the recovery-specific destination as `/auth/reset-password`.

### [medium] [CONFIRMED] Middleware redirectTo drops the original query string
File: middleware.ts:135
What happens: Middleware stores only `pathname` in `redirectTo` (`middleware.ts:74` and `middleware.ts:135`). The login page correctly copies that value into a hidden form field (`app/(auth)/login/page.tsx:38` and `app/(auth)/login/page.tsx:125` to `app/(auth)/login/page.tsx:126`), and the login action redirects to it after `signInWithPassword` (`app/(auth)/login/actions.ts:95` to `app/(auth)/login/actions.ts:98`, `app/(auth)/login/actions.ts:132` to `app/(auth)/login/actions.ts:133`). End to end propagation works for the path, but any original search params are already gone before login sees them.
User-visible effect: A protected URL such as `/alignment/new?partnerId=...` or any future dashboard deep link with query state returns to `/alignment/new` after login, losing the query-driven state.
Fix sketch: Build the preserved destination from `pathname + request.nextUrl.search` before setting `redirectTo`, then keep the existing same-site validation in the login action.

### [low] [SUSPECTED] Reset password success performs push and immediate refresh
File: app/auth/reset-password/page.tsx:108
What happens: After `updateUser({ password })` succeeds, the page calls `router.push('/dashboard')` and immediately calls `router.refresh()` (`app/auth/reset-password/page.tsx:98` to `app/auth/reset-password/page.tsx:109`). The reset page is client-only and the dashboard navigation should fetch the server-rendered dashboard with the current cookies on its own.
User-visible effect: This can create an extra route refresh around the same moment as navigation, which can show the reset page's checking/form state for a frame or add a visible snap during the move to dashboard. I did not run a browser trace, so this is suspected from the App Router call pattern rather than runtime proof.
Fix sketch: Use a single navigation, preferably `router.replace('/dashboard')`, and remove the immediate `router.refresh()` unless a measured stale-data problem requires a targeted refresh later.

### [low] [CONFIRMED] Signed-in users are redirected from /login but not from /
File: middleware.ts:125
What happens: `/` is listed as public (`middleware.ts:26` to `middleware.ts:28`), while only `/login` and `/signup` are auth routes (`middleware.ts:42`). Middleware redirects authenticated users away from auth routes only (`middleware.ts:125` to `middleware.ts:127`). The homepage server component renders the marketing page without an auth check (`app/page.tsx:24` to `app/page.tsx:62`), and its CTAs point to `/signup` and `/login` through the imported homepage/header components (`components/homepage/Hero.tsx:37` to `components/homepage/Hero.tsx:41`, `components/homepage/CTASection.tsx:36` to `components/homepage/CTASection.tsx:64`, `components/layout/Header.tsx:61` to `components/layout/Header.tsx:68`).
User-visible effect: An authenticated user visiting `/login` should be server-redirected to `/dashboard` with no login-page flash. An authenticated user visiting `/` instead sees the public homepage and auth CTAs; clicking those CTAs then bounces through middleware to dashboard. That is not a redirect loop, but it can feel like landing on the wrong page before snapping into the app.
Fix sketch: Decide the product rule for signed-in `/`. If `/` should be app entry, redirect authenticated root requests to `/dashboard` in middleware or a server component. If the public homepage should remain visible to signed-in users, make the header/CTAs auth-aware so they point directly to dashboard.

## Flow trace

Fresh visitor signup, confirmation, login, dashboard:

1. The homepage renders `Header`, `Hero`, and `CTASection` from `app/page.tsx:24` to `app/page.tsx:62`. The visible account CTAs route to `/signup` and `/login` in `components/homepage/Hero.tsx:37` to `components/homepage/Hero.tsx:41`, `components/homepage/CTASection.tsx:36` to `components/homepage/CTASection.tsx:64`, and `components/layout/Header.tsx:61` to `components/layout/Header.tsx:68`.
2. `/signup` is public in middleware (`middleware.ts:26` to `middleware.ts:35`). If a signed-in user requests `/signup`, middleware sends them to `/dashboard` (`middleware.ts:125` to `middleware.ts:127`).
3. The signup form posts to `signupAction` (`app/(auth)/signup/page.tsx:86`). The action validates fields, calls `supabase.auth.signUp`, and sets `emailRedirectTo` to `/auth/callback` (`app/(auth)/signup/actions.ts:145` to `app/(auth)/signup/actions.ts:153`).
4. If Supabase returns a session immediately, email confirmation is disabled and signup redirects to `/dashboard` (`app/(auth)/signup/actions.ts:212` to `app/(auth)/signup/actions.ts:215`). If no session is returned, the user stays on signup and sees the check-your-email message (`app/(auth)/signup/actions.ts:216` to `app/(auth)/signup/actions.ts:221`; rendered by `app/(auth)/signup/page.tsx:63` to `app/(auth)/signup/page.tsx:72`).
5. For a default PKCE/code confirmation link, `/auth/callback?code=...` exchanges the code for a session (`app/auth/callback/route.ts:96` to `app/auth/callback/route.ts:98`) and, without a safe `next`, redirects to `/login?verified=true` (`app/auth/callback/route.ts:99` to `app/auth/callback/route.ts:108`). The login page shows the verified banner (`app/(auth)/login/page.tsx:88` to `app/(auth)/login/page.tsx:96`).
6. For a custom hook token_hash signup email, the email hook builds `/auth/callback?token_hash=...&type=signup&redirect_to=...` (`app/api/auth/email/route.ts:197` to `app/api/auth/email/route.ts:203`). The callback verifies the token (`app/auth/callback/route.ts:110` to `app/auth/callback/route.ts:117`) and redirects to `next`, `redirect_to`, or `/dashboard` (`app/auth/callback/route.ts:119` to `app/auth/callback/route.ts:124`). Because signup supplies `/auth/callback` as the redirect target, this can produce the extra callback/login/dashboard bounce described above.
7. The login page reads `redirectTo` from the query (`app/(auth)/login/page.tsx:37` to `app/(auth)/login/page.tsx:40`) and submits it as a hidden input (`app/(auth)/login/page.tsx:123` to `app/(auth)/login/page.tsx:127`). The login action signs in with password and redirects to the safe `redirectTo` value or `/dashboard` (`app/(auth)/login/actions.ts:95` to `app/(auth)/login/actions.ts:98`, `app/(auth)/login/actions.ts:132` to `app/(auth)/login/actions.ts:133`).
8. `/dashboard` is protected by middleware for unauthenticated requests (`middleware.ts:130` to `middleware.ts:136`). The dashboard page also checks the current user server-side and redirects to `/login` if none is found (`app/dashboard/page.tsx:20` to `app/dashboard/page.tsx:28`), then renders `DashboardClient` for authenticated users (`app/dashboard/page.tsx:37` to `app/dashboard/page.tsx:43`).

Forgotten password reset:

1. The login page links to `/auth/forgot-password` (`app/(auth)/login/page.tsx:157` to `app/(auth)/login/page.tsx:162`). `/auth/*` is public in middleware (`middleware.ts:52` to `middleware.ts:53`).
2. The forgot-password page calls `forgotPasswordAction` on submit (`app/auth/forgot-password/page.tsx:22`, `app/auth/forgot-password/page.tsx:29` to `app/auth/forgot-password/page.tsx:41`).
3. `forgotPasswordAction` calls `resetPasswordForEmail` with `redirectTo` set to `/auth/callback?next=/auth/reset-password` (`app/(auth)/login/actions.ts:180` to `app/(auth)/login/actions.ts:185`). On success, the page shows the check-your-email state (`app/auth/forgot-password/page.tsx:83` to `app/auth/forgot-password/page.tsx:102`).
4. For a default PKCE/code recovery link, the callback exchanges the code and honors safe `next=/auth/reset-password` (`app/auth/callback/route.ts:96` to `app/auth/callback/route.ts:105`). For the custom token_hash recovery email, the email hook builds a reset URL with `type=recovery` and `redirect_to=/auth/reset-password` (`app/api/auth/email/route.ts:206` to `app/api/auth/email/route.ts:210`), and the callback verifies it and redirects to the resolved destination or the recovery fallback (`app/auth/callback/route.ts:110` to `app/auth/callback/route.ts:124`).
5. `/auth/reset-password` checks for a browser Supabase session with `getSession` and `onAuthStateChange` (`app/auth/reset-password/page.tsx:57` to `app/auth/reset-password/page.tsx:75`). Without a session it shows the expired/invalid reset-link state (`app/auth/reset-password/page.tsx:158` to `app/auth/reset-password/page.tsx:176`). With a session it shows the new-password form (`app/auth/reset-password/page.tsx:179` to `app/auth/reset-password/page.tsx:261`).
6. Submitting the reset form validates the new password, calls `supabase.auth.updateUser({ password })`, shows a success toast, then navigates to `/dashboard` and refreshes (`app/auth/reset-password/page.tsx:77` to `app/auth/reset-password/page.tsx:109`). The immediate refresh is the suspected extra flicker point noted above.
