# document-signatures audit

## Verdict
ISSUES-FOUND

## Findings

### high CONFIRMED Resolution re-analysis is blocked by the database transition guard
File: app/api/alignment/analyze/route.ts:160
What happens: After both parties submit resolution picks, the client sends users back to analysis for re-analysis (app/alignment/[id]/resolution/resolution-form.tsx:313). The analysis API accepts `resolving` alignments (app/api/alignment/analyze/route.ts:133) and tries to lock them by updating status to `analyzing` (app/api/alignment/analyze/route.ts:157-164). The app type state machine allows `resolving -> analyzing` (app/lib/types.ts:486), but the database trigger only allows `resolving -> complete` (supabase/migrations/20251110051815_init_human_alignment.sql:414) and no later migration changes that trigger.
User-visible effect: Users who resolve conflicts and need a fresh zero-conflict analysis can get stuck before the document/signature phase. The final document is reachable when the existing analysis already has zero conflicts, but the resolution loop cannot reliably produce that final zero-conflict analysis.
Fix sketch: Make the database transition guard match the app state machine by allowing `resolving -> analyzing`, or stop using `analyzing` as the lock state during re-analysis and use a separate lock column/table. Keep app/lib/types.ts, db-helpers transition validation, and the trigger in one shared contract.

### high CONFIRMED The document and sign endpoints accept unresolved `resolving` alignments
File: app/alignment/[id]/document/page.tsx:93
What happens: The document page allows any alignment whose status is `resolving` or `complete` (app/alignment/[id]/document/page.tsx:92-107). The sign API uses the same coarse gate and accepts `resolving` or `complete` (app/api/alignment/[id]/sign/route.ts:83-89). The intended zero-conflict paths exist in the analysis and resolution pages (app/alignment/[id]/analysis/page.tsx:253-260 and app/alignment/[id]/resolution/page.tsx:94-99), but neither the document page nor the signing route enforces that conflicts are actually gone.
User-visible effect: A participant can manually open `/alignment/:id/document` and sign while unresolved conflicts still exist. The generated document then includes only agreements from the analysis summary and omits unresolved conflicts from the signing decision.
Fix sketch: Add a shared server-side `assertReadyForDocument(alignmentId, round)` check used by both the document page and `/sign`. It should verify the canonical latest analysis/resolution state has zero remaining conflicts before rendering or accepting signatures.

### high CONFIRMED The canonical snapshot is per-signature, not frozen once for both signers
File: app/api/alignment/[id]/sign/route.ts:151
What happens: Each POST to `/sign` builds a new snapshot from current responses, analysis, template questions, and a fresh timestamp (app/api/alignment/[id]/sign/route.ts:123-169), then inserts that snapshot into that user's signature row (app/api/alignment/[id]/sign/route.ts:235-255). Completion checks only whether every participant has a signature for the round (app/api/alignment/[id]/sign/route.ts:262-276); it never compares snapshot hashes across signers. Separately, the displayed document is generated on the client from `/api/alignment/generate-document` (app/alignment/[id]/document/components/document-content.tsx:61-78), while the generation route returns HTML without persisting it (app/api/alignment/generate-document/route.ts:317-325).
User-visible effect: Party A and Party B can sign different hashes for the same agreement round. A template edit, analysis/response mutation, prompt skeleton change, or even the snapshot timestamp between signatures can produce divergent signed snapshots without invalidating the first signature or blocking completion.
Fix sketch: Create one immutable agreement snapshot per alignment/round before the first signature, including rendered document HTML or document sections, template/prompt versions, responses, analysis, and a single SHA-256 hash. Store signatures as references to that snapshot hash, reject signing if a different current hash would be produced, and require explicit regeneration to clear/restart signatures.

### medium CONFIRMED Document generation re-runs after every signature refresh
File: app/alignment/[id]/document/components/document-content.tsx:34
What happens: `DocumentContent` calls `generateDocument()` in a client `useEffect` and POSTs to `/api/alignment/generate-document` on mount/prop changes (app/alignment/[id]/document/components/document-content.tsx:34-89). After a successful signature, `SignatureSection` calls `router.refresh()` (app/alignment/[id]/document/components/signature-section.tsx:65-66), which re-fetches server props and causes the client document to render/generate again. The route is deterministic and does not call an AI model (app/api/alignment/generate-document/route.ts:5-9), so this is not AI token waste, but it is repeated document work and telemetry/rate-limit consumption.
User-visible effect: Party A signing, Party B signing, manual reloads, and completion refreshes can all regenerate the visible document instead of displaying the exact signed artifact. If the prompt skeleton or date changes between visits, the visible document can drift from what the first signer reviewed.
Fix sketch: Persist the generated document per alignment/round and serve it by ID/hash. Regenerate only through an explicit action that invalidates signatures, or only before the first signature when no immutable snapshot exists.

### medium CONFIRMED Concurrent double-sign attempts are not idempotent
File: app/api/alignment/[id]/sign/route.ts:219
What happens: The sign route checks for an existing signature (app/api/alignment/[id]/sign/route.ts:219-233), then inserts a new row (app/lib/db-helpers.ts:500-522). The database does prevent duplicate rows with a unique `(alignment_id, user_id, round)` constraint (supabase/migrations/20251110051815_init_human_alignment.sql:89-98), but two concurrent requests can both pass the pre-insert check. The loser hits the insert error path and the route throws a generic `Failed to create signature` (app/api/alignment/[id]/sign/route.ts:247-260). The client disables the button after state flips to signing (app/alignment/[id]/document/components/signature-section.tsx:51 and app/alignment/[id]/document/components/signature-section.tsx:183-186), but there is no server idempotency.
User-visible effect: A double-click, retry, or flaky network replay can show a signing failure even though the first request actually saved the signature. In simultaneous-sign cases, completion email sending is also keyed to `allSigned`, not to "this request performed the transition" (app/api/alignment/[id]/sign/route.ts:292-333), so duplicate completion emails are possible under race timing.
Fix sketch: Make signing idempotent. Use an insert/upsert or RPC transaction that catches the unique violation and returns the existing signature plus current `allSigned`/status. Send completion emails only when the request successfully transitions `resolving -> complete`.

### medium CONFIRMED PDF export omits the executed signatures
File: app/alignment/[id]/document/components/document-actions.tsx:28
What happens: The download action exports only the DOM node with id `alignment-document` (app/alignment/[id]/document/components/document-actions.tsx:28-48). That id is inside `DocumentContent` (app/alignment/[id]/document/components/document-content.tsx:129-132). The signature section is rendered separately after `DocumentContent` (app/alignment/[id]/document/page.tsx:153-174), so it is outside the exported node.
User-visible effect: After both parties sign, "Download Agreement" produces a PDF of the agreement body without the signature block, signer names, signature timestamps, or fully executed status.
Fix sketch: Export a wrapper that contains the document body plus the signature section, or generate the PDF server-side from the immutable agreement snapshot and signature rows.

### low CONFIRMED Dashboard does not surface the ready-to-sign status
File: app/lib/hooks/useDashboardData.ts:107
What happens: The hook comments say it uses `alignment_status_view` (app/lib/hooks/useDashboardData.ts:60-61), but it queries `alignments` directly (app/lib/hooks/useDashboardData.ts:107-120) and sets `ui_status` to the raw status (app/lib/hooks/useDashboardData.ts:129-132). `AlignmentCard` has special copy for `aligned_awaiting_signatures` ("Sign agreement") (components/dashboard/AlignmentCard.tsx:69-79) and prefers `ui_status` (components/dashboard/AlignmentCard.tsx:111-120), but the hook never supplies that derived value.
User-visible effect: A resolving alignment that is ready for signatures appears as generic resolving work on the dashboard. Clicking still eventually reaches the document through `/alignment/:id -> /resolution -> /document` when the analysis has zero conflicts, but the card copy points users to conflict resolution instead of signing.
Fix sketch: Query a corrected status view or derive `aligned_awaiting_signatures` in the hook from the same server readiness check used by the document page and sign API. Do not rely on the current view as-is without adding a zero-conflict/readiness condition.

## Flow trace

1. Document reachability: `/alignment/:id/document` authenticates, loads the alignment, and allows status `resolving` or `complete` (app/alignment/[id]/document/page.tsx:79-107). `/alignment/:id` sends every raw `resolving` alignment to `/resolution` first (app/alignment/[id]/page.tsx:36-47), and the resolution page redirects to `/document` when `analysis.summary.conflicts` is empty (app/alignment/[id]/resolution/page.tsx:94-99). The analysis page also links directly to `/document` when its parsed conflict list is empty (app/alignment/[id]/analysis/page.tsx:253-260).

2. Document generation: the server page renders `DocumentContent` (app/alignment/[id]/document/page.tsx:153-161). On the client, `DocumentContent` POSTs to `/api/alignment/generate-document` with final positions derived from `analysis.summary.agreements` (app/alignment/[id]/document/components/document-content.tsx:44-70), receives `documentHtml`, sanitizes it, and renders it into `#alignment-document` (app/alignment/[id]/document/components/document-content.tsx:77-132). The route fills a DB-managed skeleton deterministically and returns HTML/sections without persisting a document row (app/api/alignment/generate-document/route.ts:276-325).

3. Party A signs: `SignatureSection` POSTs `{ round }` to `/api/alignment/:id/sign` (app/alignment/[id]/document/components/signature-section.tsx:47-58). The route validates participant/status/round, checks no existing signature, generates a fresh canonical snapshot, inserts Party A's signature row, and counts signatures for the round (app/api/alignment/[id]/sign/route.ts:205-276). With only one signature, `allSigned` is false, status stays `resolving`, and the client calls `router.refresh()` to show Party A as signed (app/api/alignment/[id]/sign/route.ts:278-354 and app/alignment/[id]/document/components/signature-section.tsx:65-66).

4. Party B signs: Party B follows the same POST path. After B's row is inserted, `getSignatures` includes both participants for the round, so `allSigned` becomes true (app/api/alignment/[id]/sign/route.ts:262-276). If the alignment was still `resolving`, the route calls `updateAlignmentStatus(..., 'complete')` (app/api/alignment/[id]/sign/route.ts:278-289), then sends completion emails when `allSigned` is true (app/api/alignment/[id]/sign/route.ts:292-333).

5. UI after completion: B's client calls `router.refresh()` after the successful response (app/alignment/[id]/document/components/signature-section.tsx:65-66). The refreshed server page re-reads signatures for the current round (app/alignment/[id]/document/page.tsx:59-64), computes `allSigned`, renders the "Complete" header badge, the fully executed signature state, and `DocumentActions` (app/alignment/[id]/document/page.tsx:123-181). No manual reload is required for the happy path, but the refresh also re-runs client-side document generation.

6. Completed dashboard/document routing: the dashboard hook currently uses raw alignment status as `ui_status` (app/lib/hooks/useDashboardData.ts:129-132). Completed alignments therefore render as `complete`, 100 percent progress, and "View your alignment" in `AlignmentCard` (components/dashboard/AlignmentCard.tsx:31-40 and components/dashboard/AlignmentCard.tsx:69-79). Clicking the card pushes `/alignment/:id` (app/dashboard/DashboardClient.tsx:128-131), and that router redirects `complete` to `/alignment/:id/document` (app/alignment/[id]/page.tsx:46-47). The document page itself accepts `complete`, so there is no post-completion redirect fight on the happy path.
