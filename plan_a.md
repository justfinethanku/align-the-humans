## 1. Product Overview

**Human Alignment** (live at https://alignthehumans.com) is a Next.js 14 web app that helps two parties reach a documented, mutual agreement on a topic (e.g. a roommate agreement, cofounder equity split, operating agreement, marketing plan) through an AI-guided structured conversation.

Core loop: one person creates an **alignment**, defines the topic/questions, invites a partner, both parties answer independently (blind to each other's answers), Claude analyzes the responses for agreements and conflicts, and the pair iterates through AI-assisted resolution until they reach consensus and sign a final document.

## 2. Status State Machine

```
draft → active → analyzing → resolving → complete
```

Defined in `app/lib/types.ts` as `AlignmentStatus` and enforced via `VALID_STATUS_TRANSITIONS` / `isValidStatusTransition()` / `getNextValidStatuses()`. API routes and Server Actions must validate transitions through this helper rather than writing status directly.

| Status | Meaning |
|---|---|
| `draft` | Alignment created, topic/questions being configured, partner not yet fully onboarded |
| `active` | Both parties invited/joined and answering questions independently |
| `analyzing` | Both sets of responses submitted; AI is comparing them for agreements/conflicts |
| `resolving` | Conflicts identified; parties are iterating with AI-suggested compromises |
| `complete` | Agreement reached, document generated and signed |

## 3. The 5-Phase Flow (Routes)

1. **New** — `app/alignment/new/page.tsx`
   Create an alignment: pick a template (or start custom), define the topic, invite a partner.
2. **Clarity** — `app/alignment/[id]/clarity/page.tsx`
   AI-assisted topic refinement via `ClarityForm`; produces/edits the question set before both parties answer. Draft state persisted (`clarity_draft`, migration `20250120120000_add_clarity_draft_and_invite_usage.sql`).
3. **Questions** — `app/alignment/[id]/questions/page.tsx`
   Each party independently answers the question set (`questionnaire-client`). Responses are private until both submit.
4. **Waiting** — `app/alignment/[id]/waiting/page.tsx`
   Interstitial shown to a party who has submitted while waiting on their partner, or while AI analysis runs.
5. **Analysis** — `app/alignment/[id]/analysis/page.tsx`
   AI compares both response sets, detects agreements and conflicts.
6. **Resolution** — `app/alignment/[id]/resolution/page.tsx` (`resolution-form.tsx`)
   Iterative, AI-assisted conflict resolution until consensus.
7. **Document** — `app/alignment/[id]/document/page.tsx`
   Final agreement document generated, signed by both parties (with PDF export), and canonically snapshotted.

Join flow for partners: `app/join/[token]/page.tsx` (invite-link based onboarding, see §5).

### Admin

`app/admin/page.tsx`, `app/admin/alignments/page.tsx`, `app/admin/users/page.tsx`, `app/admin/prompts/page.tsx`, `app/admin/system/page.tsx` — internal dashboard gated by `profiles.is_admin` (added in `20260217140000_add_is_admin_to_profiles.sql`). Not part of the end-user flow.

## 4. API Endpoints (`app/api/`)

### Alignment core
- `POST /api/alignment/analyze` — Compare both parties' responses, detect conflicts and agreements (uses `analysisSchema`, `generateObject`).
- `POST /api/alignment/generate-questions` — AI-generate a question set from clarity context.
- `POST /api/alignment/resolve-conflicts` — Suggest compromises for identified conflicts.
- `POST /api/alignment/generate-document` — Generate the final agreement document (see also `generate-document/README.md`).
- `POST /api/alignment/get-suggestion` — AI suggestion for a specific topic/question.
- `POST /api/alignment/clarity/suggest` — AI-assisted form customization during the Clarity phase.
- `POST /api/alignment/join` — Accept an invite token and attach the joining user as a participant.

### Per-alignment resource routes (`app/api/alignment/[id]/`)
- `GET /invite` — Fetch the alignment's current active invite link (decrypts stored ciphertext to reconstruct the shareable URL).
- `POST /generate-invite` — Generate a new shareable invite token.
- `POST /regenerate-invite` — Rotate/replace an existing invite token.
- `PATCH /update` — Update alignment fields/status (validated against `VALID_STATUS_TRANSITIONS`).
- `POST /submit-resolution` — Submit a party's resolution/compromise choice during the Resolution phase.
- `POST /sign` — Record a party's signature on the final document (`alignment_signatures`, canonical snapshot).

### Auth / platform
- `POST /api/auth/email` — Auth email webhook/handler (see LIVE REALITY notes: must be wired to Supabase Auth "Send Email" hook in the dashboard, guarded by `SUPABASE_AUTH_HOOK_SECRET`).
- `GET /api/partners/search` — Search existing partners (for the Add Partner flow).
- `GET /api/admin/models` — Admin: list available AI models (Vercel AI Gateway model discovery).

All AI endpoints use the Vercel AI SDK's `generateObject()` with Zod schemas (`app/lib/schemas.ts`) rather than `streamText()`, model `anthropic('claude-sonnet-4-5-20250929')` (see `app/lib/ai-config.ts` — `AI_MODELS.{SONNET,HAIKU}`, `models.{sonnet,haiku}`, `AI_DEFAULTS`). Temperature 0.3 for analysis, higher for creative/compromise suggestions.

## 5. Invite System

Partners join via shareable invite links at `/join/[token]`.

Security model (`app/lib/invite-tokens.ts`):
- Tokens are 256-bit random values, base64url-encoded.
- Only a SHA-256 hash of the token is stored in `alignment_invitations`, plus an optional AES-256-GCM encrypted ciphertext (`INVITE_TOKEN_SECRET`) for retrieval/display purposes.
- The raw token is returned to the inviter only once, at generation time — never persisted in plaintext.
- Invite usage/expiry tracked via columns added in `20250120120000_add_clarity_draft_and_invite_usage.sql`.
- Redemption flow: `GET /join/[token]` (page) → `POST /api/alignment/join` (accept + attach participant row in `alignment_participants`).

## 6. Data Model

Core tables (Supabase/Postgres, RLS enabled on all, policies scoped via `auth.uid()`):

| Table | Purpose |
|---|---|
| `profiles` | User profile, includes `is_admin` boolean (added `20260217140000`) |
| `alignments` | The alignment record: topic, `status` (state machine), `created_by`, template reference, `clarity_draft` |
| `alignment_participants` | Join table linking users (creator + partner) to an alignment |
| `alignment_responses` | JSONB `answers` (`ResponseAnswers`, versioned, keyed by question ID) per participant |
| `alignment_analyses` | AI output: `summary` (`AnalysisSummary` — score, conflicts, agreements) and `details` (`AnalysisDetails` — full AI output, token counts) |
| `alignment_signatures` | `canonical_snapshot` (`CanonicalSnapshot` — frozen signed content with SHA-256 hash) per signer |
| `partners` | User-to-user partner relationships (for reuse across multiple alignments) |
| `templates` | Curated question-set templates; `content` is `TemplateContent` (questions array with AI hints) |
| `alignment_invitations` | Invite tokens (hashed), expiry, usage tracking |

Migrations live in `supabase/migrations/`, applied in filename-timestamp order. Key ones:
- `20251110051815_init_human_alignment.sql` — base schema + RLS
- `20251110133651_add_alignment_invitations.sql` — invite system
- `20250120120000_add_clarity_draft_and_invite_usage.sql` — clarity draft + invite usage tracking
- `20251111012029_fix_mutual_recursion_alignments.sql` — replaced participant-aware RLS policies on `alignments` with creator-only policies to fix a recursion bug between `alignments` and `alignment_participants` (⚠️ as of this writing, no follow-up migration has restored non-creator participant SELECT/UPDATE access — see known issues below)
- `20260217140000_add_is_admin_to_profiles.sql` — admin flag on `profiles`

Row-level security is enforced via two Supabase client factories (`app/lib/supabase-server.ts`): `createServerClient()` (RLS-bound, used for normal user-scoped reads/writes) and `createAdminClient()` (service-role, bypasses RLS, server-only — used sparingly for privileged operations).

## 7. Known Issues / Launch Blockers (as of 2026-07-01)

- **RLS partner lockout**: `alignments` table RLS currently only grants SELECT/UPDATE to `created_by = auth.uid()`. A partner who is a row in `alignment_participants` but not the creator cannot read/update the parent `alignments` row directly via RLS-bound queries. Whether this breaks the live app depends on whether `app/lib/db-helpers.ts` reads alignments via `createServerClient()` (broken for partners) or `createAdminClient()` (masks the bug). Needs a follow-up migration restoring participant-aware SELECT/UPDATE policies without reintroducing the original recursion bug.
- **Auth email hook**: whether Supabase's "Send Email" auth hook is wired to `/api/auth/email` must be confirmed manually in the Supabase dashboard (Authentication → Hooks). If unwired, Supabase falls back to default transactional emails silently. If wired with a mismatched `SUPABASE_AUTH_HOOK_SECRET`, auth emails fail outright.
- **No admin user provisioned**: `profiles.is_admin` exists but no row has it set to `true` in production; an admin must be promoted manually before the admin dashboard is usable.
- **`prompts` table**: not present in the live schema as of this writing; if the admin prompt-management UI expects one, a migration is still needed (or prompts remain hardcoded in `app/lib/prompts.ts`).
