Human Alignment backend fit analysis • May 2026

Convex is probably the better fit if you are open to rebuilding Human Alignment.

Human Alignment is a two-person AI-guided decision workflow: people join an alignment, answer independently, wait for each other, review AI analysis, resolve conflicts, and produce a signed agreement. The product is fundamentally collaborative, stateful, and event-driven. That shape lines up strongly with Convex’s reactive TypeScript model.

Recommendation If the goal is the best long-term architecture for this product, I would seriously consider a Convex rebuild. If the goal is the fastest path to a first release from the current codebase, keep Supabase and ship the existing app.

What The App Is About

Product

Structured agreement between people

The app helps two parties make decisions together by separating independent thinking from negotiation. It works for cofounders, roommates, couples, business partners, teams, and other high-stakes conversations.

Workflow

Five-phase alignment flow

Create an alignment, clarify the topic, answer questions independently, analyze responses, resolve conflicts, then generate a final agreement with signatures.

AI role

Facilitator, not chat toy

AI generates questions, gives suggestions, compares responses, identifies conflicts, helps with resolution, and generates the final document.

My read: this is a collaborative workflow product, not a database-heavy ledger product. That is the core reason Convex may be a better foundation.

Current Architecture

Why Convex Is A Good Fit

1. The app is naturally reactive

The waiting page, dashboard, partner join state, response submission state, analysis status, and resolution progress all benefit from live updates. Convex makes query subscriptions the default instead of a special-case hook.

2. The data model is workflow-shaped

Alignments, participants, answers, analyses, drafts, and signatures are app-state documents with relationships. They do not require heavy SQL analytics to deliver the core product.

3. TypeScript mutations match the product

Actions like create alignment, join invite, save answer, submit response, start analysis, save analysis, submit resolution, and sign document would read naturally as Convex mutations/actions.

4. AI workflows could be cleaner

AI question generation, analysis, conflict suggestions, and document generation could run as Convex actions with stored status and results. The UI could subscribe to progress without polling or manual refreshes.

Reasons Against Convex

1. The current app already has a working backend

The app already has Supabase Auth, server/browser clients, RLS, typed generated database types, invite token storage, Realtime hooks, and helper functions. Replacing that is a real rebuild, not a config change.

2. Auth and privacy are central

Partner responses must stay private until both parties submit. Supabase RLS is a strong fit for enforcing participant-scoped access. Convex can do authorization, but you would need to rebuild and audit those rules in application functions.

3. Signed agreements favor boring durability

Final documents, signatures, snapshots, hashes, and audit logs benefit from Postgres-style durability, relational constraints, and long-term inspectability.

4. A rebuild is only worth it if the product becomes more live

If Human Alignment stays mostly step-by-step, request/response, and document-centered, Supabase is already good enough. Convex becomes clearly better when the app behaves like a shared live workspace.

Best Architecture If Starting From Zero

My Recommendation

If rebuilding is on the table: choose Convex

For a fresh version of Human Alignment, I would build around Convex queries, mutations, and actions. The app’s core interactions are live workflow state, not complex SQL reporting.

If shipping the existing app fast: keep Supabase

The current Supabase foundation is coherent. Keeping it is the practical path if your immediate goal is release, validation, and learning from users.

Direct answer: Convex is a better fit for the ideal version of Human Alignment if you want it to feel like a living shared workspace. Supabase is still reasonable for the existing implementation, especially around auth, RLS, and signed archival records.

If You Stay With Supabase

1. Store resolution drafts server-side

The resolution form currently leans on localStorage. Move drafts into Supabase so a user can switch devices, recover work, and avoid losing sensitive thinking.

2. Tighten workflow mutations

Make create alignment, add owner participant, join invite, submit response, start analysis, submit resolution, and sign document feel like named product operations rather than scattered table updates.

3. Expand realtime narrowly

You already have a realtime hook. Use it for response submission, partner join, analysis completion, and resolution updates. Keep polling as a fallback.

4. Add launch-readiness admin views

Track incomplete alignments, failed AI calls, stuck analysis states, invite usage, signed documents, and user drop-off by phase.

5. Keep RLS boring and explicit

This app handles private partner answers. Supabase RLS should stay front and center. Audit policies before release.

6. Reduce stale model assumptions

The repo names specific Claude models from late 2025. Before release, verify model names, pricing, and AI Gateway config against current provider docs.

Rebuild Decision Plan

Step 1

Define the product you want

Decide whether Human Alignment should be a guided form workflow or a live collaborative workspace. That choice matters more than the database choice.

Step 2

If form workflow: keep Supabase

If users mostly move through sequential pages, answer questions privately, and generate a document, the current architecture is enough.

Step 3

If live workspace: rebuild on Convex

If both parties should see presence, progress, drafts, comments, AI status, shared resolution state, and notifications in real time, Convex is the cleaner foundation.

Step 4

Prototype the hardest Convex slice

Build only one slice first: create an alignment, invite a partner, answer independently, and watch submission status update live. If that feels dramatically simpler, the rebuild has evidence.

Step 5

Keep the signed document model boring

Even in a Convex rebuild, treat final signed agreements, hashes, and exports as a deliberately stable archival layer.

Step 6

Do not do a half-migration

A split brain between Supabase workflow state and Convex live state will slow you down. Pick one canonical workflow backend.

Bottom Line

Convex is a strong fit for Human Alignment because the product is collaborative, stateful, reactive, and TypeScript-heavy. If you are open to a clean rebuild and you want the app to feel alive while two people move through a decision together, Convex is probably the better foundation.

Supabase remains a good fit if the product stays closer to a secure step-by-step form workflow with final signed documents. The backend choice should follow the product shape: live shared workspace points to Convex; secure sequential agreement builder points to Supabase.
